## Console Logging & Debugging

### Automatic Log Capture

**All console output is automatically captured** by the SAM SDK for debugging purposes.

Every time you use `console.log()`, `console.warn()`, `console.error()`, `console.info()`, or `console.debug()`, the log is:
1. **Displayed in the browser DevTools** (normal behavior preserved)
2. **Captured in memory** (last 100 logs kept)
3. **Available to the user** via a "Copy Logs" button in the preview pane

### User Access to Logs

The user can click the **"Copy Logs"** button in the app preview header to copy all captured console logs to their clipboard. They can then paste these logs into the chat to share them with the App Agent for debugging assistance.

### Best Practices for Console Logging

**CRITICAL**: Keep console output MINIMAL during normal operation.

✅ **DO**:
- Use `console.error()` for critical errors that need immediate attention
- Log important state changes sparingly (e.g., "User authenticated", "Data loaded")
- Add verbose logging TEMPORARILY when actively debugging an issue
- Remove debug logging after the issue is resolved

❌ **DON'T**:
- Log every render cycle or frequent events
- Log verbose information during normal operation
- Use console.log() as a permanent debugging tool
- Spam the console with unnecessary messages

### Why Minimal Logging Matters

1. **Memory**: Only the last 100 logs are kept, so excessive logging pushes out useful error messages
2. **Performance**: Excessive logging can slow down your app
3. **Usability**: When the user copies logs, they want to see errors and important events, not noise

### Examples

**Good** (Minimal, meaningful logging):
```typescript
// Log errors
try {
  await doSomething();
} catch (error) {
  console.error("Failed to do something:", error);
}

// Log important events
const handleLogin = async () => {
  const user = await authenticateUser();
  console.log("User authenticated:", user.id);
};
```

**Bad** (Excessive, noisy logging):
```typescript
// DON'T log every render
function MyComponent() {
  console.log("MyComponent rendering");  // ❌ Too frequent
  return <div>Hello</div>;
}

// DON'T log trivial operations
const handleClick = () => {
  console.log("Button clicked");  // ❌ Unnecessary
  setCount(count + 1);
  console.log("Count updated");   // ❌ Unnecessary
};
```

### Debugging Workflow

When actively debugging an issue:
1. Add verbose logging temporarily to track down the problem
2. Use `console.error()` for errors, `console.warn()` for warnings
3. Test the fix
4. **Remove the verbose logging** before completing the task
5. Keep only essential error logging

### Browser DevTools

Remember: All logs still appear in the browser's DevTools console. Users can:
- Open DevTools (F12) for real-time log viewing
- Use DevTools filtering and search features
- Access the full console API (not just the captured logs)

The "Copy Logs" button is a convenience feature for sharing logs with the App Agent, but it doesn't replace DevTools for interactive debugging.

## SAM SDK Documentation

### Overview

The SAM SDK (`@sam/sdk`) is a **production-ready TypeScript library** that enables your app to integrate with the SAM platform via `postMessage` communication with the parent frame.

**IMPORTANT**: This is NOT a mock or test SDK. All API calls communicate with the real SAM platform:
- `SAM.agents.call()` → Calls actual SAM agents in the platform
- `SAM.storage` → Reads/writes real persistent storage
- `SAM.artifacts` → Uploads/downloads real files
- `SAM.ui` → Gets real theme state from parent

The SDK provides APIs for:

- **Agent Calling**: Invoke other SAM agents from your app
- **Storage**: Persistent, app-scoped key-value data storage
- **Artifacts**: File upload/download
- **UI Integration**: Theme detection and dark mode support

### Installation

The SAM SDK is already installed in this template. Import it in your components:

```typescript
import { SAM } from '@sam/sdk'
import type { Theme } from '@sam/sdk'
```

### Initialization

**IMPORTANT**: Always call `SAM.ready()` before using any SDK features.

```typescript
import { useEffect } from 'react'
import { SAM } from '@sam/sdk'

function App() {
  useEffect(() => {
    const init = async () => {
      // Wait for SDK to establish connection with parent frame
      await SAM.ready()

      // Now safe to use all SDK features
      const theme = SAM.ui.getTheme()
      // ...
    }

    init()
  }, [])
}
```

#### 2. SAM.agents - Agent Discovery and Calling

Discover available agents and call them from your app.

**Methods:**
```typescript
SAM.agents.list(): Promise<AgentInfo[]>
SAM.agents.call(agentName: string, options: AgentCallOptions): Promise<AgentCallResult>
```

**Types:**
```typescript
interface AgentInfo {
  id: string                        // Unique agent identifier
  name: string                      // Agent name
  description?: string              // Agent description
  version?: string                  // Agent version
  capabilities?: string[]           // Agent capabilities
}

interface AgentCallOptions {
  prompt: string                    // Required: The prompt/instruction for the agent
  context?: Record<string, any>     // Optional: Additional context data
  stream?: boolean                  // Optional: Enable streaming response
  files?: File[]                    // Optional: Attach documents/images to agent call
  timeout?: number                  // Optional: Timeout in milliseconds (default: 300000 / 5 minutes)
  onText?: (text: string) => void   // Optional: Callback for streaming text chunks
  onStatus?: (status: string) => void // Optional: Callback for status updates
  onArtifact?: (artifact: any) => void // Optional: Callback when an artifact is created
}

interface AgentCallResult {
  response: string                  // The agent's response
  artifacts?: string[]              // Array of artifact IDs (if any)
  metadata?: Record<string, any>    // Additional response metadata
}
```

**IMPORTANT: How to Work with Files and Agents**

When working with files and agent calls, follow these critical guidelines:

1. **Always attach files to agent calls** - Use the `files` parameter:
   ```typescript
   // ✅ CORRECT: Agent receives the file and can process it
   await SAM.agents.call('Agent', {
     prompt: 'Analyze this image',
     files: [imageFile]
   })

   // ❌ WRONG: Agent doesn't receive the file, only sees text
   const artifactId = await SAM.artifacts.upload(imageFile)
   await SAM.agents.call('Agent', {
     prompt: 'Analyze this image',
     context: { artifactId }  // Agent can't see the file!
   })
   ```

2. **Files must be in the message** - Agents can only see files that are included in the message parts. Uploading an artifact separately and referencing it by ID does NOT make the file available to the agent.

3. **Alternative: Reference by name** - If you've already uploaded a file to artifacts in a previous conversation turn, you can reference it by filename:
   ```typescript
   // Upload in first message
   await SAM.agents.call('Agent', {
     prompt: 'Store this image for later',
     files: [imageFile]  // Creates artifact: chart.png
   })

   // Reference in subsequent message (same session)
   await SAM.agents.call('Agent', {
     prompt: 'Apply a filter to chart.png'  // Reference by name
   })
   ```

4. **Artifact upload returns full result** - If you do use `SAM.artifacts.upload()` directly, it returns an object with upload details:
   ```typescript
   const result = await SAM.artifacts.upload(file)
   console.log(result.artifactId)  // "artifact://gateway/user/session/myfile.pdf?version=0"
   console.log(result.sessionId)   // Session ID for this upload
   console.log(result.filename)    // "myfile.pdf"
   console.log(result.size)        // File size in bytes
   console.log(result.mimeType)    // "application/pdf"
   ```

5. **When to use SAM.artifacts.upload()** - Only use direct artifact upload when:
   - You need to upload a file WITHOUT calling an agent
   - You're building a file management UI
   - You need the upload metadata (sessionId, size, etc.) for your application logic

**Common Mistake to Avoid:**

```typescript
// ❌ DON'T DO THIS - Agent can't see the file
async function processImage(image: File) {
  // Upload separately
  const result = await SAM.artifacts.upload(image)

  // Then call agent with just the artifact ID
  await SAM.agents.call('ImageProcessor', {
    prompt: 'Apply sepia filter',
    context: { artifactId: result.artifactId }  // ❌ Agent doesn't have access to this!
  })
}

// ✅ DO THIS INSTEAD - Agent receives the file
async function processImage(image: File) {
  await SAM.agents.call('ImageProcessor', {
    prompt: 'Apply sepia filter',
    files: [image]  // ✅ Agent can see and process the file
  })
}
```

**Session Management:**

By default, the SDK automatically maintains a persistent session across ALL operations (agent calls AND artifact uploads). This ensures:
- Uploaded files remain accessible in subsequent operations
- Conversation history is preserved
- Agents remember context from previous interactions
- Session continuity works in BOTH directions

**The SDK automatically tracks sessions across:**
- Agent calls → subsequent agent calls
- Agent calls → subsequent artifact uploads
- Artifact uploads → subsequent agent calls
- Artifact uploads → subsequent artifact uploads

```typescript
// Example 1: Multi-turn conversation with persistent session

  // First call: Upload file and ask question
  const result1 = await SAM.agents.call('DataAnalyst', {
    prompt: 'Analyze this dataset',
    files: [csvFile]
  });
  console.log('Session ID:', result1.sessionId); // e.g., "web-session-abc123"

  // Second call: Follow-up question (automatically uses same session)
  const result2 = await SAM.agents.call('DataAnalyst', {
    prompt: 'What were the top 3 trends?'
  });
  console.log('Session ID:', result2.sessionId); // Same session ID

  // Third call: Reference file by name from first call
  const result3 = await SAM.agents.call('DataAnalyst', {
    prompt: 'Create a chart from the data.csv file'
  });
  // Agent can access data.csv from the first call's session
}

// Example 2: Agent call THEN artifact upload (session continuity maintained)
```

**Explicit Session Control:**

For advanced use cases, you can explicitly manage sessions across BOTH agent calls AND artifact uploads:

```typescript
// Example 1: Multiple independent sessions (run-based mode)
async function independentSessions() {
  // Session 1: Upload file for Project A
  const upload1 = await SAM.artifacts.upload(fileA, { sessionId: null });
  const session1 = upload1.sessionId;

  // Session 2: Upload file for Project B (separate session)
  const upload2 = await SAM.artifacts.upload(fileB, { sessionId: null });
  const session2 = upload2.sessionId;

  // Agent call for Project A (uses session 1)
  await SAM.agents.call('Analyzer', {
    prompt: 'Analyze the first file',
    sessionId: session1
  });

  // Agent call for Project B (uses session 2)
  await SAM.agents.call('Analyzer', {
    prompt: 'Analyze the second file',
    sessionId: session2
  });
}

// Example 2: Shared session across agents and uploads
async function sharedSession() {
  let sessionId: string;

  // Start with agent call
  const result1 = await SAM.agents.call('DocumentParser', {
    prompt: 'I will upload some documents',
    sessionId: null  // Create new session
  });
  sessionId = result1.sessionId;

  // Upload files to SAME session
  await SAM.artifacts.upload(pdfFile, { sessionId });
  await SAM.artifacts.upload(csvFile, { sessionId });

  // Continue conversation in same session
  await SAM.agents.call('DataVisualizer', {
    prompt: 'Create charts from the uploaded data',
    sessionId  // All operations share same session
  });
}

// Example 3: Multi-workspace app with isolated sessions
async function multiWorkspace() {
  const workspaces = {
    project1: null as string | null,
    project2: null as string | null
  };

  // Create separate session for each workspace
  const upload1 = await SAM.artifacts.upload(file1, { sessionId: null });
  workspaces.project1 = upload1.sessionId;

  const upload2 = await SAM.artifacts.upload(file2, { sessionId: null });
  workspaces.project2 = upload2.sessionId;

  // Each workspace operates in isolation
  await SAM.agents.call('Agent', {
    prompt: 'Work on project 1',
    sessionId: workspaces.project1
  });

  await SAM.agents.call('Agent', {
    prompt: 'Work on project 2',
    sessionId: workspaces.project2
  });
}

// Example 4: Reset to new session at any time
async function resetSession() {
  // Previous operations used session A

  // Force new session for upload
  const result = await SAM.artifacts.upload(file, { sessionId: null });
  // result.sessionId is a NEW session, persistent session NOT updated

  // Continue with new session
  await SAM.agents.call('Agent', {
    prompt: 'Start fresh task',
    sessionId: result.sessionId
  });
}
```

**Important Session Notes:**

1. **Automatic by default**: SDK tracks session ID internally across ALL operations (agent calls AND uploads)
2. **Files are session-scoped**: Uploaded files are only accessible within their session
3. **Explicit control**: Pass `sessionId` option to override automatic session management
4. **Force new session**: Pass `sessionId: null` to create a new session (run-based mode)
5. **Return value**: Every operation returns `sessionId` so you can store and reuse it
6. **Multi-session apps**: Advanced apps can maintain multiple isolated sessions for different workflows

**Working with Artifacts Returned by Agents (RECOMMENDED PATTERN)**

When an agent generates artifacts (images, documents, etc.), the **BEST PRACTICE** is to capture them from the `onArtifact` callback during streaming, rather than relying on the `result.artifacts` array.

**Why?**
- `onArtifact` provides artifact details in real-time as they're created
- You can display artifacts immediately (progress indication)
- You get both the artifact URI and metadata
- More reliable than waiting for the final response

**Example: Image editing app with real-time artifact display**
```typescript
function ImageEditor() {
  const [inputImage, setInputImage] = useState<File | null>(null)
  const [outputImageUrl, setOutputImageUrl] = useState<string | null>(null)
  const [processing, setProcessing] = useState(false)

  const handleEditImage = async (instructions: string) => {
    if (!inputImage) return

    setProcessing(true)
    setOutputImageUrl(null)

    // Store the most recent artifact from callback
    let resultArtifact = null

    try {
      const result = await SAM.agents.call('ImageEditingAgent', {
        prompt: `${instructions}\n\nReturn the edited image using artifact_return.`,
        files: [inputImage],
        stream: true,
        onStatus: (status) => {
          console.log('Status:', status)
        },
        onArtifact: (artifact) => {
          // ✅ BEST PRACTICE: Store artifact object from callback
          console.log('Artifact received:', artifact.name)
          resultArtifact = artifact
        }
      })

      // Download and display the result
      if (resultArtifact) {
        // Simply call .download() on the artifact - it handles everything!
        const blob = await resultArtifact.download()
        const imageUrl = URL.createObjectURL(blob)
        setOutputImageUrl(imageUrl)
      } else {
        console.warn('No artifacts generated by agent')
      }
    } catch (error) {
      console.error('Image editing failed:', error)
    } finally {
      setProcessing(false)
    }
  }

  return (
    <div>
      <input
        type="file"
        accept="image/*"
        onChange={(e) => setInputImage(e.target.files?.[0] || null)}
      />
      <button onClick={() => handleEditImage('Apply sepia filter')} disabled={!inputImage || processing}>
        {processing ? 'Processing...' : 'Edit Image'}
      </button>
      {outputImageUrl && <img src={outputImageUrl} alt="Result" />}
    </div>
  )
}
```

**Key Points:**
- The artifact object received in `onArtifact` has a built-in `.download()` method
- No need to extract URIs or handle different cases - just call `.download()`
- Works whether the artifact has bytes embedded or needs to be fetched from a URI
- Simple, foolproof API that "just works"

**Alternative: Using result.artifacts array**

You can also use `result.artifacts`, which contains artifact objects accumulated during the call. However, this requires waiting for the entire call to complete:

```typescript
const result = await SAM.agents.call('ImageEditingAgent', {
  prompt: 'Apply sepia filter',
  files: [inputImage]
})

// result.artifacts is an array of artifact objects with .download() method
if (result.artifacts && result.artifacts.length > 0) {
  const lastArtifact = result.artifacts[result.artifacts.length - 1]

  // Just call .download() on the artifact object!
  const blob = await lastArtifact.download()
  const imageUrl = URL.createObjectURL(blob)
  setOutputImageUrl(imageUrl)
}
```

**Key Differences:**

| Approach | Pros | Cons |
|----------|------|------|
| **`onArtifact` callback** | Real-time updates, immediate display, shows progress | Slightly more code |
| **`result.artifacts` array** | Simpler code | Only available after completion, no progress indication |

**Recommendation:** Use `onArtifact` for better user experience, especially when:
- Processing takes more than a few seconds
- You want to show progress/status updates
- You're building interactive applications

## Building and Versioning

### Build Script

After making changes to your app, you should build and version it using the provided script:

```bash
# Build and increment patch version (default)
./build_and_version.sh patch

# Build and increment minor version
./build_and_version.sh minor

# Build and increment major version
./build_and_version.sh major
```

**What the script does:**
1. Reads the current version from `VERSION` file (starts at 0.0.0 if not present)
2. Increments the version based on semver rules:
   - **patch**: Bug fixes and small changes (0.0.1 → 0.0.2)
   - **minor**: New features, backwards compatible (0.1.0 → 0.2.0)
   - **major**: Breaking changes (1.0.0 → 2.0.0)
3. Runs `npm run build` to compile the app
4. If build succeeds:
   - Creates a git commit with message "Release vX.Y.Z"
   - Creates a git tag "vX.Y.Z"
   - Updates the `VERSION` file with version metadata
5. If build fails: Exits without committing (no broken versions)

### Version File

The `VERSION` file contains metadata about the current build:

```json
{
  "version": "1.2.3",
  "timestamp": 1702345678000,
  "commit": "abc123",
  "increment": "minor"
}
```

This file is used by the deployment system to track which version is currently deployed.

### When to Use Each Increment Type

- **patch** (default): Most changes - bug fixes, UI tweaks, small improvements
- **minor**: New features that don't break existing functionality
- **major**: Breaking changes, major redesigns, significant API changes

**Note**: The App Agent will automatically call `build_and_version.sh patch` after making changes. You can tell it to use `minor` or `major` if you're making significant changes.

### Deployment

Once you've built and versioned your app, click the **Deploy** button in the UI to:
1. Copy the built files to the deployment directory
2. Update the production pointer to the new version
3. Make the app available at the deployed URL

The deployment system maintains version history, allowing rollbacks if needed.
